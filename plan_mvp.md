# AI-Копилот Продажник -- План MVP для команды

---

## Что мы строим

Реалтайм-система, которая во время телефонного звонка помогает менеджеру закрывать сделки. Система встраивается **между** существующей телефонией и CRM клиента -- как мост, через который проходит звонок и данные.

**Что мы хотим получить от MVP:**

- **Выстроить рабочий пайплайн** -- сквозную структуру от входящего звонка до подсказки на экране менеджера. Каждый элемент (распознавание речи, движок интентов, генерация подсказок, CRM-интеграция) потом можно улучшать независимо
- **Проверить юзабельность в реальности** -- насколько подсказки действительно помогают менеджеру, не мешают ли, успевают ли вовремя
- **Оценить сложность подключения клиента** -- сколько усилий стоит онбординг: настройка SIP, маршрутизация, заведение базы знаний
- **Понять стоимость одного звонка** -- расход на STT, LLM, инфраструктуру; можно ли строить на этом бизнес
- **Заложить архитектуру на будущее** -- чтобы каждый модуль можно было заменить или масштабировать, не переписывая всё с нуля

**Из чего состоит:**

| Компонент | Что это | Технологии |
|---|---|---|
| **Бэкенд (сервер)** | Принимает аудиопоток звонка, распознаёт речь, определяет намерения клиента, генерирует подсказки, общается с CRM | Python (FastAPI), Yandex SpeechKit, OpenAI/YandexGPT |
| **Телефонный прокси** | Пропускает звонок насквозь и отдает нам аудио в два канала (клиент отдельно, менеджер отдельно) | Asterisk (бесплатная АТС) |
| **Фронтенд (виджет менеджера)** | Показывает текст разговора, данные клиента, подсказки ИИ, кнопки действий. Можно встраивать в CRM там, где это возможно | React / JavaScript, WebSocket |
| **CRM-адаптер** | Подгружает данные клиента при звонке, записывает итоги после звонка. Отдельный модуль для каждой CRM | REST API (Bitrix24, amoCRM, др.) |

### Как система подключается

**Со стороны телефонии** -- к любому SIP-провайдеру. Система встает между провайдером (Mango, Zadarma, Sipuni, любой SIP) и менеджером. Клиенту **не нужно менять** номер или провайдера.

**Со стороны CRM** -- к любой CRM через адаптеры. Ядро системы (распознавание речи, ИИ, подсказки) и виджет менеджера -- **общие**, со своим фронтенд-окном, которое можно встраивать в CRM где это возможно. Для каждой CRM пишется свой адаптер.

## Архитектура: Путь 1 -- Asterisk SIP-прокси (основной)

Для клиентов с **внешним SIP-провайдером** (Mango, Zadarma, Sipuni и т.д.):

```
Клиент звонит
      |
      v
Телефонный провайдер (Mango / любой SIP)
      |
      v
+---------------------+
|  Наш сервер-прокси   |  Пропускает звонок насквозь,
|     (Asterisk)       |  но копирует звук в два потока:
+----------+-----------+  голос клиента и голос менеджера
      |         |
      |         +---> Менеджер слышит клиента
      |               (софтфон / SIP Connector Bitrix24)
      v
+------------------------------------------------------+
|                    Наш бэкенд                         |
|                                                       |
|  1. Распознавание речи      (Yandex SpeechKit)        |
|     Два раздельных потока -> текст с пометками        |
|     "Клиент" / "Менеджер"                             |
|                                                       |
|  2. Интеграция с CRM                                  |
|     (REST API Bitrix24 / amoCRM / любая CRM)          |
|     - При старте: подгружает данные клиента           |
|     - После звонка: пишет саммари и задачи            |
|                                                       |
|  3. Движок определения намерений                      |
|     (на базе промптов и базы знаний компании)         |
|     - Анализирует реплики клиента                     |
|     - Классифицирует: дорого, подумаю,                |
|       конкурент, готов купить и т.д.                   |
|                                                       |
|  4. Генератор подсказок      (OpenAI / YandexGPT)     |
|     - Что сказать                                     |
|     - Закрывающий вопрос                              |
|     - Следующее действие                              |
|                                                       |
|  5. Отправка в виджет        (WebSocket)              |
+---------------------------+---------------------------+
                            |
                            v
              +-------------------------+
              |  Виджет менеджера        |
              |  (встраивается в CRM     |
              |   там где возможно)      |
              |                          |
              |  - Текст разговора       |
              |  - Карточка клиента      |
              |  - Подсказки ИИ          |
              |  - Кнопки действий       |
              +--------------------------+
```

### Маршрутизация звонка: откуда и куда

1. **Номер для клиентов** -- берётся у SIP-провайдера (например, Mango). Это тот номер, на который звонят клиенты.
2. **Asterisk принимает звонок** от провайдера и перенаправляет менеджеру.
3. **Менеджер отвечает** одним из способов:
   - Через **софтфон** (MicroSIP на компьютере), подключенный к нашему Asterisk
   - Через **SIP Connector Bitrix24** -- звонок приходит прямо в интерфейс Bitrix24
   - Через **перенаправление на мобильный** -- Asterisk делает исходящий звонок на мобильный менеджера через SIP-провайдера (то есть через тот же Mango/Zadarma) и соединяет два плеча
4. **Исходящие звонки** -- тот же Asterisk, но маршрут обратный: менеджер инициирует звонок через софтфон или SIP Connector, Asterisk перехватывает и соединяет с клиентом через SIP-провайдера. Два аудиопотока и весь пайплайн подсказок работают так же, как при входящем.

### Два раздельных потока и перебивания

Asterisk отдаёт нам два **раздельных** аудиопотока: голос клиента и голос менеджера. Нам **не нужно** определять "кто говорит" -- мы и так это знаем. Даже если они перебивают друг друга -- текст не "смешивается".

> **`directmedia=no`** -- настройка: звук всегда проходит через наш сервер. Задержки: ~1-5 мс (менее 3% от обычной мобильной задержки). Незаметно.

### Объект сессии звонка (CallSession)

Каждый активный звонок = одна сессия (`CallSession`) -- in-memory объект, объединяющий: два аудиопотока, WebSocket-соединение с виджетом, контекст клиента из CRM, историю реплик, текущий интент, таймстампы. Все компоненты бэкенда работают в рамках одной сессии -- это основа, без которой данные звонка разрозненны.

## А что если у клиента Bitrix24 "всё в одном"?

Если клиент использует **встроенную облачную телефонию Bitrix24** (арендованный номер внутри Битрикса), звук не покидает инфраструктуру Bitrix24. Вставить наш Asterisk в цепочку нельзя.

Но есть решение -- **Путь 2: свой WebRTC-софтфон внутри Bitrix24**.

## Архитектура: Путь 2 -- WebRTC-софтфон внутри Bitrix24

Bitrix24 позволяет встраивать **собственный WebRTC-клиент** (софтфон) как приложение. Это работает через механизм `PAGE_BACKGROUND_WORKER` -- невидимый iframe, который загружается на каждой странице Битрикса.

```
Клиент звонит
      |
      v
Телефония Bitrix24 (встроенная или внешняя)
      |
      v
+----------------------------------+
|  Bitrix24 (браузер менеджера)     |
|                                   |
|  +-----------------------------+  |
|  | PAGE_BACKGROUND_WORKER      |  |  <-- Наше приложение
|  | (невидимый iframe)          |  |      в фоне
|  |                             |  |
|  | - Свой WebRTC-клиент        |  |  Мы контролируем
|  | - Перехват MediaStream      |  |  аудиопоток
|  | - Отправка аудио на бэкенд  |  |
|  +-------------+---------------+  |
|                |                  |
|  +-------------v---------------+  |
|  | Виджет менеджера (iframe)   |  |  <-- Наше UI
|  | Транскрипт + Подсказки      |  |      внутри Bitrix24
|  +-----------------------------+  |
+----------------------------------+
                |
                v
      Наш бэкенд (тот же)
```

**Как это работает:** наше приложение встраивается в Bitrix24 как невидимый фоновый скрипт. При звонке оно перехватывает аудиопоток прямо в браузере и отправляет на наш бэкенд. Дальше -- тот же пайплайн: распознавание речи, интенты, подсказки.

> **Примечание**: Путь 2 сложнее Пути 1 (нужно писать свой WebRTC-клиент), но позволяет работать **без внешнего SIP-провайдера** и **без отдельного сервера Asterisk**. Техническая детализация и все ссылки на документацию Bitrix24 -- в файле `technical_reference.md`.

Но я бы не завязывался только на битрикс, если они внезапно добавят в свой BitrixGPT настраиваемые действия или еще какой то инструментарий для подсказок в реальном времени, то приложение резко станет неактуальным. Поэтому я бы сделал ставку на то, что мы можем быть не привязаны к какой-то одной CRM, а работать с любым клиентом, у которого есть внешний SIP-транк и софтфон/мобильный.

## Реалистичная скорость подсказок

| Элемент | Время |
|---|---|
| Передача аудио-чанка | ~50 мс |
| Распознавание речи (final) | ~500-1500 мс |
| LLM: генерация подсказки | ~800-2000 мс |
| Передача в UI | ~50 мс |
| **Итого** | **~1.5-3.5 сек** |

Реалистично: **2-4 секунды** от конца фразы клиента до подсказки. Менеджер получает подсказку, пока формулирует ответ.

> В будущем можно ускорить, анализируя **partial-транскрипты** (неоконченные фразы) и прогнозируя намерение на базе контекста предыдущего диалога -- но это уже не для MVP.

## Отказоустойчивость и edge-кейсы

**Падение Asterisk.** Если наш прокси-сервер недоступен, SIP-провайдер должен быть настроен на fallback-маршрут: звонок идёт напрямую менеджеру. Подсказки не работают, но связь не теряется. Клиент не должен остаться без телефонии из-за наших проблем.

**Как это реализовать в Mango Office:**

В ВАТС Mango Office есть три механизма, которые вместе дают нужный failover:

1. **Схема распределения звонков с условием "если не ответили".** В личном кабинете Mango Office -> настройки ВАТС -> схема распределения звонков. На номер, привязанный к нашему SIP-транку, добавляем блок "если не ответили" (настраивается таймаут, например 5 секунд). Если наш Asterisk не взял звонок за это время -- вызов перенаправляется на резервный адресат: внешний номер (мобильный менеджера), другой SIP-транк, конкретный сотрудник или группа сотрудников.

2. **SIP OPTIONS (проверка доступности).** Mango Office отправляет SIP OPTIONS запросы к нашему серверу для проверки его статуса. Если сервер не отвечает -- транк помечается как недоступный, и ВАТС автоматически перестает отправлять звонки по этому маршруту. Таймаут проверки настраивается на стороне нашего Asterisk (параметр `qualify`).

3. **Резервный SIP-транк.** В схеме маршрутизации Mango можно указать два SIP-транка: основной (наш Asterisk) и резервный (напрямую к оборудованию менеджера или обратно к Mango ВАТС). Если основной транк недоступен (по SIP OPTIONS) -- звонки идут через резервный.

**Рекомендуемая конфигурация для MVP:**
- Основной маршрут: входящий звонок -> наш Asterisk -> менеджер (с подсказками)
- Fallback (5 сек без ответа): звонок -> напрямую на софтфон/мобильный менеджера (без подсказок, но связь работает)
- Настраивается целиком в личном кабинете Mango Office, не требует кода с нашей стороны

**Отказ внешних сервисов.** SpeechKit, LLM или CRM API могут упасть или не ответить вовремя. Принцип: звонок и виджет продолжают работать, отказавший компонент отключается с уведомлением:
- STT недоступен -- виджет показывает "транскрипция временно недоступна", подсказки не генерируются
- LLM не ответил -- подсказка не отображается (без ошибки на экране)
- CRM не ответила -- карточка клиента пустая, остальное работает

**Reconnect виджета.** Если менеджер обновил страницу или пропал интернет -- виджет автоматически переподключается по WebSocket и получает текущее состояние звонка (последние реплики, активную подсказку, данные клиента).

**Аутентификация WebSocket.** Подключение виджета -- через JWT-токен, привязанный к `call_id` + `user_id`. Без валидного токена -- отказ подключения.

## Тестовая среда MVP

Для разработки и тестирования MVP используем связку **Mango Office** (SIP-провайдер) + **Bitrix24** (CRM). Это даёт нам: реальный SIP-транк с гибкой маршрутизацией и failover, богатый REST API для CRM-интеграции, тестовый портал Bitrix24, и возможность проверить оба пути (Asterisk и WebRTC) на одной среде.

## Этапы разработки MVP

### Этап 1 -- Инфраструктура и сервер

Получаем все доступы, поднимаем сервер, устанавливаем Asterisk, настраиваем SIP-транк от провайдера, настраиваем маршрут на софтфон/SIP Connector менеджера.

> **Контрольная точка**: тестовый звонок проходит через наш сервер. Клиент звонит на номер провайдера -- менеджер видит входящий и слышит клиента. В логах сервера видно: звонок принят, два аудиопотока передаются на бэкенд.

---

### Этап 2 -- Распознавание речи + данные из CRM

Подключаем Yandex SpeechKit. Два аудиоканала превращаются в текст с пометками "Клиент" / "Менеджер". Параллельно подключаем **интеграцию с CRM**: при входящем звонке бэкенд автоматически ищет контакт по номеру и достаёт данные сделки.

> **Контрольная точка**: во время живого тестового звонка в браузере появляется текст разговора с указанием, кто говорит. Рядом -- карточка клиента, подгруженная из CRM. Текст обновляется в реальном времени.

---

### Этап 3 -- ИИ-движок подсказок (MVP-минимум)

Для MVP делаем базовый вариант: бэкенд отправляет текст реплики + контекст клиента в LLM, получает подсказку. Без сложных конечных автоматов и скоринга -- это всё можно будет улучшать после MVP.

Минимальный набор интентов: "дорого", "подумаю", конкурент, "не сейчас", сигнал к покупке.

Подсказка содержит: что сказать + закрывающий вопрос + следующее действие.

Ограничение: не более 1 подсказки за X секунд -- иначе менеджер будет читать подсказки и терять нить диалога, или начнет их игнорировать.

**База знаний компании (MVP-формат):** простой JSON/YAML-файл с FAQ, типичными возражениями и рекомендованными ответами. Клиент заполняет шаблон в Google Sheets -- мы конвертируем. На работающем MVP оценим: достаточный объём данных, оптимальный формат, необходимость UI для редактирования.

> **Контрольная точка**: при тестовом звонке, когда "клиент" говорит "дорого" -- на экране через 2-4 секунды появляется подсказка. Подсказки не спамят -- приходят только на значимые реплики.

---

### Этап 4 -- Виджет менеджера + полная интеграция

Собираем виджет:
- **Текст разговора** -- live-транскрипт ("Клиент" / "Менеджер")
- **Карточка клиента** -- имя, компания, сделка, стадия, бюджет (из CRM)
- **Подсказки ИИ** -- цель, фраза, закрывающий вопрос (обновляются динамически)
- **Скоринг закрытия** -- вероятность + что мешает
- **Кнопки действий** -- "Отправить КП", "Создать сделку", "Назначить встречу"

По завершении звонка:
- Саммари автоматически в CRM
- Задача follow-up
- Менеджер выбирает результат звонка

**Чеклист подключения клиента:** пошаговая инструкция онбординга -- от получения SIP-кредов провайдера до первого тестового звонка с подсказками.

> **Контрольная точка**: полный цикл работает от начала до конца. Менеджер принимает звонок -- видит текст, данные клиента, подсказки в реальном времени. После звонка -- в CRM автоматически появляется саммари и задача. Виджет открывается в отдельном окне рядом с CRM (или как iframe внутри, если CRM позволяет).
